## 3. ForeignKey a Fondo: Más Allá de lo Básico

### 3.1 Lo que ocurre en la base de datos

Cuando definimos:

```python
# tienda/models.py
from django.db import models

class Categoria(models.Model):
    nombre = models.CharField(max_length=100)

    class Meta:
        verbose_name = 'Categoría'
        verbose_name_plural = 'Categorías'
        ordering = ['nombre']

    def __str__(self):
        return self.nombre


class Producto(models.Model):
    nombre  = models.CharField(max_length=200)
    precio  = models.DecimalField(max_digits=10, decimal_places=2)
    stock   = models.IntegerField(default=0)
    activo  = models.BooleanField(default=True)

    categoria = models.ForeignKey(
        Categoria,
        on_delete=models.CASCADE,
        related_name='productos',
    )

    class Meta:
        verbose_name = 'Producto'
        ordering = ['nombre']

    def __str__(self):
        return f"{self.nombre} (${self.precio})"
```

Django ejecuta este SQL al aplicar la migración:

```sql
-- Tabla del modelo padre (se crea primero — no tiene dependencias)
CREATE TABLE "tienda_categoria" (
    "id"     bigint NOT NULL PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    "nombre" varchar(100) NOT NULL
);

-- Tabla del modelo hijo (se crea después — necesita que la anterior exista)
CREATE TABLE "tienda_producto" (
    "id"           bigint NOT NULL PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    "nombre"       varchar(200) NOT NULL,
    "precio"       numeric(10,2) NOT NULL,
    "stock"        integer NOT NULL DEFAULT 0,
    "activo"       boolean NOT NULL DEFAULT TRUE,
    "categoria_id" bigint NOT NULL
        REFERENCES "tienda_categoria" ("id")
        DEFERRABLE INITIALLY DEFERRED
);

-- Índice automático en la columna de clave foránea
CREATE INDEX "tienda_producto_categoria_id_idx"
    ON "tienda_producto" ("categoria_id");
```

**Tres decisiones automáticas que Django tomó por nosotros:**

**1. Renombró el campo:** `categoria` en Python → `categoria_id` en la BD.
En nuestra clase Python escribimos `categoria`, pero en la base de datos esa columna se llama `categoria_id`. Django aplica esta convención automáticamente para dejar en claro que esa columna es solo un *número* que apunta a otra tabla.

**2. Agregó la restricción `REFERENCES`:**
Esta línea del SQL es fundamental:
```sql
"categoria_id" bigint NOT NULL
    REFERENCES "tienda_categoria" ("id")
```
`REFERENCES` le dice a la base de datos: *"el valor que guardes aquí DEBE existir como `id` en la tabla `tienda_categoria`"*. Si alguien intenta guardar un producto con `categoria_id = 9999` y esa categoría no existe, la base de datos rechaza la operación. Esta protección funciona incluso si alguien se conecta directo a la BD y se salta completamente el código Python.

**3. Creó un índice automático en `categoria_id`:**
Cada vez que hacemos un JOIN o un filtro por categoría (`filter(categoria_id=3)`), la base de datos necesita buscar rápidamente todas las filas que tienen ese valor. Sin índice, revisaría fila por fila toda la tabla. Con el índice, va directo. Django lo crea automáticamente en toda columna de clave foránea porque es una de las columnas más consultadas de cualquier tabla.

### 3.3 El campo en Python vs en la base de datos

Aquí hay una diferencia importante que confunde a muchos developers al principio: **la base de datos guarda un número, Python muestra un objeto**.

Cuando la BD almacena un producto, su columna `categoria_id` solo contiene el número `1`. Nada más. Pero cuando Django trae ese producto al código Python, hace la traducción: convierte ese `1` en el objeto `Categoria` completo, con todos sus campos. Esta magia ocurre de forma transparente.

Entender esto permite elegir la forma más eficiente de acceder a los datos:

```python
producto = Producto.objects.get(id=1)
# ↑ Django trajo el producto desde la BD.
# En su memoria ya tiene: id=1, nombre='Laptop...', categoria_id=1
# Pero todavía NO trajo la categoría — solo tiene el número.

producto.categoria
# ↑ Aquí Django detecta que necesitas el objeto Categoria completo.
# Dispara una consulta SQL adicional: SELECT * FROM tienda_categoria WHERE id=1
# Devuelve: <Categoria: Electrónica>

producto.categoria.nombre
# ↑ El objeto ya está en memoria (lo trajo la línea anterior).
# Solo accede al atributo — sin nueva consulta.
# Devuelve: "Electrónica"

producto.categoria_id
# ↑ Este es el número crudo que vino con el producto desde el principio.
# No necesita ir a ningún lado — ya está en memoria.
# Devuelve: 1
```

**¿Cuándo conviene usar cada uno?**

| Necesito... | Usar | ¿Consulta extra? |
|:------------|:-----|:-----------------|
| El nombre, descripción u otro campo de la categoría | `producto.categoria.nombre` | ✅ Sí (a menos que uses `select_related`) |
| Solo el ID numérico de la categoría | `producto.categoria_id` | ❌ No — ya está en memoria |

> La raíz del problema N+1 que vimos en Clase 1 está exactamente aquí: cada vez que se accede a `producto.categoria` sin haber usado `select_related`, Django hace una consulta a la BD. En un loop de 200 productos, esas consultas se acumulan.
>
> *Fuente: Django Documentation. (2024). Database access optimization — Use foreign key values directly.*

## 4. on_delete: Cinco Decisiones de Negocio

El parámetro `on_delete` es **obligatorio** en Django y expresa una regla de negocio crítica: *¿qué le pasa a los productos si elimino su categoría?* No hay una respuesta universal — depende de cada proyecto.

### El escenario de NebulaShop

Imaginemos que la categoría **"Telescopios"** (id=3) tiene 47 productos. Un administrador intenta eliminarla. ¿Qué debería pasar?

---

### 4.1 `CASCADE` — Todo el árbol cae

```python
categoria = models.ForeignKey(Categoria, on_delete=models.CASCADE)
```

**Lo que ocurre en la BD:**
```sql
DELETE FROM tienda_categoria WHERE id = 3;
-- Automáticamente: DELETE FROM tienda_producto WHERE categoria_id = 3;
```

Los 47 productos se eliminan sin posibilidad de recuperación.

**Cuándo es la elección correcta:**
- El objeto hijo no tiene valor sin el padre.
- Ejemplos: comentarios de una publicación, ítems de un carrito de compras, imágenes de un álbum.

**Cuándo es peligroso:**
- Datos históricos (pedidos realizados, facturas, registros de auditoría).
- Información con valor independiente del padre.

---

### 4.2 `SET_NULL` — El producto sobrevive sin categoría

```python
categoria = models.ForeignKey(
    Categoria,
    on_delete=models.SET_NULL,
    null=True,   # Obligatorio: la columna debe aceptar NULL
    blank=True,  # Para que formularios no lo exijan
)
```

**Lo que ocurre en la BD:**
```sql
DELETE FROM tienda_categoria WHERE id = 3;
-- Automáticamente: UPDATE tienda_producto SET categoria_id = NULL
--                  WHERE categoria_id = 3;
```

Los 47 productos siguen existiendo, pero su `categoria_id` queda `NULL`. Aparecerán como "Sin categoría" hasta que un administrador los reclasifique.

**Cuándo usarlo:**
- La relación es opcional o secundaria.
- El objeto hijo tiene valor independiente del padre.
- Ejemplos: un artículo de blog cuyo autor fue eliminado, un producto cuya categoría fue reorganizada.

**Error típico:** Olvidar agregar `null=True` al usar `SET_NULL`. Django levantará un error al hacer la migración:

```
django.db.utils.IntegrityError: NOT NULL constraint failed: tienda_producto.categoria_id
```

---

### 4.3 `PROTECT` — Django bloquea la eliminación del padre

```python
categoria = models.ForeignKey(Categoria, on_delete=models.PROTECT)
```

**Lo que ocurre cuando alguien intenta borrar la categoría:**
```python
categoria = Categoria.objects.get(id=3)
categoria.delete()

# Django lanza:
# django.db.models.deletion.ProtectedError:
# ("Cannot delete some instances of model 'Categoria' because they are
# referenced through protected foreign keys: 'Producto.categoria'", {<...>})
```

Django *ni siquiera* envía el `DELETE` a la base de datos. Lo bloquea en Python antes.

**Cuándo usarlo:**
- Datos críticos que nunca deberían perderse accidentalmente.
- Ejemplos: una cuenta bancaria con movimientos, un cliente con pedidos procesados, un proveedor con contratos activos.

**Buena práctica:** Usar `PROTECT` como opción por defecto cuando no estás seguro. Es mejor tener un error controlado que perder datos en producción.

---

### 4.4 `SET_DEFAULT` — El producto se reasigna automáticamente

```python
# Asumiendo que la Categoría "General" siempre tiene id=1
categoria = models.ForeignKey(
    Categoria,
    on_delete=models.SET_DEFAULT,
    default=1,  # ID de la categoría "General" o "Sin clasificar"
)
```

**Lo que ocurre en la BD:**
```sql
DELETE FROM tienda_categoria WHERE id = 3;
-- Automáticamente: UPDATE tienda_producto SET categoria_id = 1
--                  WHERE categoria_id = 3;
```

**Cuándo usarlo:**
- Existe una categoría "comodín" que siempre debe existir.
- Se quiere mantener la trazabilidad pero con una categoría genérica de respaldo.

**Riesgo:** Si el registro con el `id` del `default` también es eliminado antes, la base de datos rechazará la operación con un error de integridad referencial.

---

### 4.5 `DO_NOTHING` — Delegas la responsabilidad a la BD

```python
categoria = models.ForeignKey(Categoria, on_delete=models.DO_NOTHING)
```

Django no hace ningún manejo especial. Si la BD tiene restricciones configuradas (como `ON DELETE RESTRICT` por defecto en PostgreSQL), la operación fallará a nivel de base de datos con un `IntegrityError`.

**Cuándo usarlo:**
- Modelos que mapean tablas existentes donde la BD ya tiene su propia lógica configurada.
- Casos donde se trabaja con bases de datos heredadas con procedimientos almacenados propios.

---

### Tabla de Decisión: ¿Cuál usar?

| Situation | Opción recomendada |
|:----------|:-------------------|
| El hijo no tiene sentido sin el padre | `CASCADE` |
| El hijo puede existir sin categoría | `SET_NULL` + `null=True` |
| No quiero que accidentalmente borren el padre | `PROTECT` |
| Hay una categoría "General" de respaldo | `SET_DEFAULT` |
| La BD ya tiene su propia lógica | `DO_NOTHING` |
| No sé qué elegir | `PROTECT` (falla visible > pérdida silenciosa) |

> *Fuente: Django Software Foundation. (2024). ForeignKey.on_delete — Model field reference.*

## 5. Consultas Inversas, QuerySet y related_name

### 5.0 ¿Qué es un QuerySet?

Antes de hablar de dirección inversa, necesitamos tener muy claro un concepto que aparece constantemente: el **QuerySet**.

Un QuerySet es el resultado de pedirle datos al ORM. No es una lista de Python común. Es un objeto inteligente que:
- **Describe** lo que queremos buscar (pero todavía NO fue a la BD).
- **Puede encadenarse** con filtros adicionales sin ejecutar consultas extras.
- **Se evalúa** (ejecuta realmente el SQL) solo cuando algo necesita los datos: un loop, un `list()`, una impresión.

```python
# Un QuerySet es una "promesa de búsqueda", no datos reales todavía
productos = Producto.objects.filter(activo=True)   # ← aún no fue a la BD
productos = productos.filter(precio__gte=100)       # ← sigue sin ir a la BD
productos = productos.order_by('nombre')            # ← sigue sin ir a la BD

for p in productos:   # ← AQUÍ Django va a la BD y ejecuta TODO junto en una sola consulta
    print(p.nombre)
```

Esto lo vimos en Clase 1. Lo que importa hoy es entender que **la relación inversa de una FK también devuelve QuerySets**, y por eso tenemos todos los mismos métodos disponibles.

---

### 5.1 De hijo a padre (dirección natural)

Esta dirección ya la conocemos. Django nos permite acceder al objeto padre directamente desde el hijo:

```python
producto = Producto.objects.get(id=1)

categoria = producto.categoria          # → <Categoria: Telescopios>
nombre    = producto.categoria.nombre   # → "Telescopios"
```

---

### 5.2 ¿Qué es related_name y para qué sirve?

Si tenemos el objeto padre (`categoria`), ¿cómo llegamos a todos sus hijos (`productos`)? Ahí entra `related_name`.

Cuando definimos la `ForeignKey` en el modelo `Producto`, estamos diciendo: *"Producto apunta a Categoria"*. Pero Django, por diseño, también crea la posibilidad de ir en sentido contrario: desde `Categoria` hacia todos sus `Productos`.

El **`related_name`** es el nombre que nosotros le damos a ese camino inverso.

```python
# Así lo definimos en el modelo:
class Producto(models.Model):
    categoria = models.ForeignKey(
        Categoria,
        on_delete=models.CASCADE,
        related_name='productos',  # ← Este nombre es el que podemos usar desde Categoria
    )
```

**Sin `related_name` definido**, Django genera uno automático usando la regla `nombre_del_modelo_set`:
```python
categoria.producto_set.all()   # ← Django lo genera solo, pero es poco expresivo
```

**Con `related_name='productos'`**, nosotros controlamos el nombre:
```python
categoria.productos.all()      # ← Más claro, describe el negocio
```

Ambos hacen exactamente lo mismo. Solo cambia el nombre del atributo con el que accedemos.

---

### 5.3 De padre a hijo (dirección inversa) — el QuerySet inverso

Desde una instancia de `Categoria`, podemos navegar a todos sus productos. Y como el resultado es un **QuerySet**, tenemos todos los métodos del ORM disponibles:

```python
telescop = Categoria.objects.get(nombre='Telescopios')
# ↑ Traje una categoría específica de la BD

# --- Ejemplos de consulta sobre sus productos ---

todos       = telescop.productos.all()
# ↑ QuerySet con todos los productos de Telescopios
# SQL: SELECT * FROM tienda_producto WHERE categoria_id = 3

en_stock    = telescop.productos.filter(stock__gt=0)
# ↑ Solo los que tienen stock disponible

mas_caros   = telescop.productos.filter(precio__gte=500).order_by('-precio')
# ↑ Los de 500 o más, ordenados de mayor a menor precio

cuantos     = telescop.productos.count()
# ↑ Ejecuta SELECT COUNT(*) — no trae los objetos, solo el número

tiene_alguno = telescop.productos.exists()
# ↑ Ejecuta SELECT 1 LIMIT 1 — la forma más rápida de saber si hay al menos uno
```

El resultado de `telescop.productos` es un QuerySet exactamente igual que el de `Producto.objects.filter(...)`. No hay diferencia técnica — solo partimos desde un camino distinto.

---

### 5.4 Filtros que cruzan la relación con `__`

El doble guión bajo `__` nos permite cruzar relaciones desde cualquier punto del ORM. Es una de las herramientas más potentes de Django:

```python
# Desde Producto, filtrando por campo de Categoria
Producto.objects.filter(categoria__nombre='Telescopios')
# SQL generado: SELECT * FROM tienda_producto
#              INNER JOIN tienda_categoria ON ...
#              WHERE tienda_categoria.nombre = 'Telescopios'

# Ir aún más profundo si hay más niveles de relación
Producto.objects.filter(categoria__nombre__istartswith='T')
# → Productos cuya categoría empiece con 'T' (sin importar mayúsculas)

# Desde Categoria, filtrando por campo de sus Productos
Categoria.objects.filter(productos__stock__gt=0).distinct()
# → Categorías que tienen al menos un producto con stock mayor a 0
# → .distinct() es obligatorio para evitar categorías duplicadas en el resultado
```

> **¿Por qué `.distinct()`?** Cuando cruzamos hacia la relación inversa, la base de datos genera una fila por cada *producto* que coincide, no por cada *categoría*. Si "Telescopios" tiene 5 productos en stock, aparece 5 veces. `.distinct()` elimina los duplicados.
>
> *Fuente: Django Documentation. (2024). Making queries — Spanning relationships.*

---

### 5.5 Dos ForeignKey al mismo modelo: related_name obligatorio

Si un modelo tiene **dos o más `ForeignKey` apuntando al mismo modelo**, Django no puede generar automáticamente dos nombres inversos distintos. Hay que definir `related_name` explícitamente en ambas:

```python
# ❌ Sin related_name → Django lanza error: "Reverse accessor clashes"
class Envio(models.Model):
    origen  = models.ForeignKey('Ciudad', on_delete=models.PROTECT)
    destino = models.ForeignKey('Ciudad', on_delete=models.PROTECT)

# ✅ Con related_name diferente en cada FK
class Envio(models.Model):
    origen  = models.ForeignKey(
        'Ciudad',
        on_delete=models.PROTECT,
        related_name='envios_salientes',   # ciudad.envios_salientes.all()
    )
    destino = models.ForeignKey(
        'Ciudad',
        on_delete=models.PROTECT,
        related_name='envios_entrantes',   # ciudad.envios_entrantes.all()
    )
```


## 6. OneToOneField: Extender Sin Romper

### 6.1 ¿Cuándo y por qué queremos "extender" un modelo?

Esta es la pregunta que hay que responder antes de ver el código.

Imagina que NebulaShop quiere guardar datos extra de sus clientes: una foto de perfil, puntos de lealtad acumulados, la ciudad donde viven. La primera idea podría ser: *¿y por qué no los agrego directamente al modelo `User` de Django?*

La respuesta es que el `User` de Django **no es nuestro modelo**. Lo provee el framework para manejar autenticación, login, sesiones y permisos. Si lo modificamos directamente:

- Podemos romper la autenticación.
- Perdemos la compatibilidad con apps de terceros que esperan el `User` original.
- En futuras versiones de Django, migrar puede volverse un caos.

Entonces la solución es crear **nuestro propio modelo** que vive aparte y se conecta al `User` con un `OneToOneField`. Cada usuario tendrá exactamente un perfil. Cada perfil pertenece exactamente a un usuario. Son dos tablas separadas que se mueven juntas.

---

### 6.2 ¿En qué se diferencia de ForeignKey?

Técnicamente, `OneToOneField` es un `ForeignKey` con una restricción `UNIQUE` que Django agrega automáticamente en la columna de la base de datos. Esa restricción garantiza que ningún `User` pueda tener dos perfiles.

```python
# ForeignKey: MUCHOS productos pueden apuntar a UNA categoría
# La columna categoria_id en Producto NO tiene UNIQUE
# → Muchos productos pueden tener categoria_id = 1

# OneToOneField: UN perfil apunta a UN usuario. Solo uno. Nunca dos.
# La columna usuario_id en PerfilCliente SÍ tiene UNIQUE
# → Si intentas crear un segundo perfil para el mismo User → error en la BD
```

---

### 6.3 Implementación: el modelo PerfilCliente

```python
# tienda/models.py
from django.db import models
from django.contrib.auth.models import User


class PerfilCliente(models.Model):
    # Un PerfilCliente existe exactamente PARA un User.
    # Si se elimina el User, se elimina su perfil también (CASCADE).
    usuario = models.OneToOneField(
        User,
        on_delete=models.CASCADE,
        related_name='perfil',
        # ↑ Esto nos permite acceder al perfil desde el user con: user.perfil
    )
    biografia      = models.TextField(blank=True)
    puntos_lealtad = models.IntegerField(default=0)
    ciudad         = models.CharField(max_length=100, blank=True)
    foto_url       = models.URLField(blank=True)

    class Meta:
        verbose_name        = 'Perfil de Cliente'
        verbose_name_plural = 'Perfiles de Clientes'

    def __str__(self):
        return f"Perfil de {self.usuario.username}"
```

**¿Qué genera Django en la base de datos?**

```sql
CREATE TABLE "tienda_perfilcliente" (
    "id"             bigint NOT NULL PRIMARY KEY ...,
    "usuario_id"     integer NOT NULL UNIQUE   -- ← UNIQUE: impide dos perfiles por user
        REFERENCES "auth_user" ("id"),
    "biografia"      text NOT NULL,
    "puntos_lealtad" integer NOT NULL DEFAULT 0,
    "ciudad"         varchar(100) NOT NULL,
    "foto_url"       varchar(200) NOT NULL
);
```

Observa que la tabla `auth_user` ya existía — la creó Django cuando hicimos `migrate` por primera vez. Nuestro `PerfilCliente` simplemente hace referencia a ella sin tocarla.

---

### 6.4 Navegar la relación en el código

**Del perfil al usuario (dirección del campo):**
```python
perfil = PerfilCliente.objects.get(id=1)

perfil.usuario              # → <User: juangarcia>  (objeto User completo)
perfil.usuario.email        # → "juan@nebulashop.com"
perfil.usuario.first_name   # → "Juan"
```

**Del usuario al perfil (usando related_name='perfil'):**
```python
user = User.objects.get(username='juangarcia')

user.perfil                    # → <PerfilCliente: Perfil de juangarcia>
user.perfil.puntos_lealtad     # → 150
user.perfil.ciudad             # → "Santiago"
```

---

### 6.5 El riesgo: el perfil puede no existir

A diferencia de `ForeignKey` donde el hijo siempre apunta a algo, en `OneToOneField` la relación inversa puede no existir. Si un `User` fue creado antes de implementar el `PerfilCliente`, simplemente no tiene perfil.

```python
# ❌ Código frágil: si el usuario no tiene perfil, esto lanza una excepción
user = User.objects.get(username='juangarcia')
puntos = user.perfil.puntos_lealtad
# → RelatedObjectDoesNotExist: User has no perfilcliente.

# ✅ Código defensivo: siempre manejar el caso donde el perfil no existe
try:
    puntos = user.perfil.puntos_lealtad
except PerfilCliente.DoesNotExist:
    puntos = 0

# Alternativa con hasattr:
if hasattr(user, 'perfil'):
    puntos = user.perfil.puntos_lealtad
else:
    puntos = 0
```

---

### 6.6 Crear el perfil automáticamente con señales

En proyectos reales hay un problema práctico: cada vez que se registra un usuario nuevo, alguien tiene que crear su `PerfilCliente`. Si lo hacemos manualmente en cada view de registro, es fácil olvidarlo en algún flujo (registro por admin, por API, por OAuth).

Django tiene un sistema de **señales** (`signals`) que permite ejecutar código automáticamente cuando ocurre un evento en el ORM. En este caso, queremos ejecutar código *después de que se guarda un User nuevo*.

```python
# tienda/signals.py
from django.db.models.signals import post_save
# ↑ post_save: señal que se dispara DESPUÉS de que se guarda cualquier objeto

from django.contrib.auth.models import User
from django.dispatch import receiver
from .models import PerfilCliente


@receiver(post_save, sender=User)
# ↑ Este decorador dice: "escucha el evento post_save del modelo User"

def crear_perfil_al_registrar(sender, instance, created, **kwargs):
    # sender   → el modelo que disparó la señal (User)
    # instance → el objeto User específico que fue guardado
    # created  → True si fue CREADO, False si fue ACTUALIZADO
    if created:
        PerfilCliente.objects.create(usuario=instance)
        # ↑ Solo al crear (no al actualizar) → creamos el perfil
```

Para que Django cargue el archivo de señales al iniciar:

```python
# tienda/apps.py
from django.apps import AppConfig

class TiendaConfig(AppConfig):
    name = 'tienda'

    def ready(self):
        import tienda.signals  # ← Django carga el archivo, registrando el listener
```

Con esto, sin importar desde dónde se cree el `User` (admin, formulario, API, testing), su `PerfilCliente` se crea automáticamente.

> Las señales son un tema que profundizaremos en una clase futura. Por ahora, lo importante es entender que **existen** y cuál es el problema que resuelven.
>
> *Fuente: Django Documentation. (2024). Signal dispatching — post_save.*

## 7. Herramientas Modernas de Performance y Seguridad

Escribir código que funciona no es suficiente. En un entorno profesional — ya sea una PYME o una empresa grande — el código debe ser **medible, seguro y auditable**. Estas son las herramientas que usan los equipos reales hoy.

### 7.1 Diagnóstico Visual en el Navegador: Django Debug Toolbar

**¿Qué es?** Una barra lateral que se inyecta en el navegador durante desarrollo y muestra en tiempo real todo lo que ocurre en cada request.

```bash
pip install django-debug-toolbar
```

**¿Qué muestra?**

| Panel | Información |
|:------|:------------|
| SQL | Cada consulta ejecutada, su duración y si hay duplicados (N+1 visible a simple vista) |
| Cache | Hits y misses del sistema de caché |
| Templates | Qué templates se renderizaron y cuánto tardó cada uno |
| Signals | Qué señales se dispararon y quiénes las escucharon |
| Tiempo total | Desglose de la latencia del request completo |

Es la primera herramienta que debe instalarse en cualquier proyecto Django nuevo. Basta con mirar el panel SQL para ver si hay un N+1: si en una página simple aparecen 80 consultas SQL, hay un problema.

> *Fuente: Django Debug Toolbar.* [https://django-debug-toolbar.readthedocs.io](https://django-debug-toolbar.readthedocs.io)

### 7.2 Profiling Avanzado para APIs: Django Silk

**¿Qué es?** Un profiler más completo que Debug Toolbar. Es especialmente útil para APIs REST donde no hay templates ni barra lateral visible en el navegador.

```bash
pip install django-silk
```

**¿Qué ofrece que Debug Toolbar no tiene?**

- **Call graph:** Un gráfico visual de qué funciones llamaron a qué, y cuánto tiempo tomó cada una.
- **Historial persistente:** Guarda los requests en la BD — puedes comparar el tiempo de ayer vs hoy.
- **Replay de requests:** Vuelves a ejecutar un request guardado para ver si mejoró luego de un cambio en el código.
- **Ordenamiento por consultas:** Lista todos los endpoints ordenados por cantidad de SQL generado — encuentras los peores casos en segundos.

```
Panel de Silk: http://localhost:8000/silk/
→ Lista de todos los requests del día
→ Cada request con su tiempo total, cantidad de queries y duración de cada query
→ Call graph interactivo por función Python
```

> *Fuente: Django Silk.* [https://github.com/jazzband/django-silk](https://github.com/jazzband/django-silk)

### 7.3 Seguridad Estática del Código: Bandit

**¿Qué es?** Un analizador estático de seguridad para Python. Lee el código fuente (sin ejecutarlo) y detecta patrones conocidos de vulnerabilidades.

```bash
pip install bandit

# Analizar toda la carpeta del proyecto
bandit -r ./miproyecto
```

**¿Qué detecta?**

| Vulnerabilidad | Ejemplo que detecta |
|:---------------|:--------------------|
| SQL injection | `cursor.execute("SELECT * FROM" + variable)` — concatenación directa |
| Contraseñas hardcodeadas | `PASSWORD = "admin123"` en el código fuente |
| Uso de `eval()` | `eval(user_input)` — ejecución de código arbitrario |
| Conexiones inseguras | HTTP sin TLS en llamadas a servicios externos |
| `mark_safe()` en Django | Marcar HTML como seguro sin sanitizarlo antes |

Bandit clasifica cada hallazgo por **severidad** (HIGH / MEDIUM / LOW) y **confianza** en el diagnóstico. Los equipos profesionales lo integran en el pipeline de CI/CD para que ningún pull request con vulnerabilidades altas pueda mergearse.

> *Fuente: PyCQA Bandit.* [https://bandit.readthedocs.io](https://bandit.readthedocs.io)

### 7.4 Seguridad de Dependencias: pip-audit y Safety

### 7.6 Monitoreo en Producción: Sentry

**¿Qué es?** Una plataforma de monitoreo que captura errores y problemas de rendimiento en tiempo real, directamente en producción.

**¿Por qué cambia la dinámica de un equipo?** Sin Sentry, el equipo se entera de un error cuando el cliente llama. Con Sentry, ven el error con stack trace completo, la versión del código donde ocurrió, y cuántos usuarios fueron afectados — antes de que nadie se queje.

```bash
pip install sentry-sdk
```

```python
# settings.py
import sentry_sdk
from sentry_sdk.integrations.django import DjangoIntegration

sentry_sdk.init(
    dsn="https://tu-dsn@sentry.io/proyecto",
    integrations=[DjangoIntegration()],
    traces_sample_rate=0.1,   # Captura el 10% de las transacciones para performance
    send_default_pii=False,   # No envía datos personales a Sentry
)
```

**Lo que Sentry registra automáticamente:**
- Cada excepción no capturada con su traceback completo.
- Tiempo de respuesta promedio por endpoint.
- Consultas SQL lentas que superan un umbral configurable.
- Alertas (ej: si el mismo error ocurre más de 100 veces en 5 minutos).

> *Fuente: Sentry Django Integration.* [https://docs.sentry.io/platforms/python/integrations/django/](https://docs.sentry.io/platforms/python/integrations/django/)

### 7.7 IA para Seguridad: Corgea y GitHub Copilot Security

Dos tendencias que el mercado está adoptando rápidamente:

**Corgea — IA que detecta Y corrige vulnerabilidades:**
- Escanea el repositorio completo con inteligencia artificial.
- Detecta patrones de vulnerabilidad (inyección SQL, XSS, CSRF mal configurado).
- **Genera automáticamente un pull request** con la corrección sugerida y una explicación del riesgo.
- Clasifica las vulnerabilidades por prioridad de negocio, no solo por severidad técnica.

**GitHub Copilot con Security Review:**
- Analiza cada pull request antes del merge, directamente en GitHub.
- Señala líneas con patrones de vulnerabilidad conocidos.
- Sugiere el código corregido en el mismo hilo de comentario del PR.

> Estas herramientas no reemplazan el juicio del desarrollador. Son amplificadores de capacidad — reducen el tiempo que toma encontrar problemas, pero el desarrollador sigue siendo quien decide si la solución sugerida es correcta en su contexto.

### 7.8 Calidad de Código y SAST Avanzado: SonarCloud / SonarQube

**¿Qué es?** La suite más adoptada en equipos profesionales para análisis estático de código. Va más allá de buscar vulnerabilidades — analiza **calidad**, **mantenibilidad** y **deuda técnica**.

**Dos variantes:**

| | SonarCloud | SonarQube |
|:--|:-----------|:----------|
| **Hosting** | SaaS en la nube (sin infraestructura) | Self-hosted — en tu servidor |
| **Precio** | Gratuito para repos públicos | Open source (Community Edition gratuita) |
| **Ideal para** | Startups y equipos pequeños | Empresas con requisitos de datos propios |
| **Integración** | GitHub, GitLab, Azure DevOps nativamente | Jenkins, GitLab CI, cualquier CI |

**¿Qué detecta en proyectos Python/Django?**

- SQL injection, XSS, CSRF mal configurado (alineado con OWASP Top 10 y CWE)
- Code smells: código duplicado, funciones demasiado largas, complejidad ciclomática alta
- Cobertura de tests: muestra qué porcentaje del código está cubierto
- Bugs probables: acceso a variables nulas, condiciones que nunca se evalúan correctamente

**Quality Gates — la clave para CI/CD:**

El concepto más potente de Sonar es el **Quality Gate**: un conjunto de condiciones que el código **debe cumplir** para que el pipeline continúe. Si no se cumple, el merge o el deploy se bloquea automáticamente.

```yaml
# Ejemplo de Quality Gate en SonarCloud para un proyecto Django:
# ✅ Ningún nuevo bug crítico
# ✅ Cobertura de tests en nuevo código ≥ 80%
# ✅ Ningún "security hotspot" nuevo sin revisar
# ✅ Duplicación de código nuevo < 3%
```

Si alguna condición falla → el pipeline se rompe → el PR no puede mergearse.

### 7.9 Análisis Semántico: Semgrep

**¿Qué es?** Una herramienta de análisis estático que entiende el **flujo de datos** dentro del código. A diferencia de Bandit (que busca patrones de texto), Semgrep entiende la semántica del código.

**¿Qué lo hace diferente?**

Consideremos este ejemplo:

```python
# Bandit NO lo detecta como peligroso (parece escapado)
nombre = request.GET.get('nombre')
valor_escapado = nombre.strip()
cursor.execute(f"SELECT * FROM users WHERE nombre = '{valor_escapado}'")

# Semgrep SÍ lo detecta: hace tracking del flujo de datos
# desde request.GET (fuente de datos no confiable)
# hasta cursor.execute (punto de inyección)
# aunque haya un .strip() en el medio
```

Semgrep tiene **reglas específicas para Django** que conocen el framework por dentro: detecta datos que vienen de `request.GET`, `request.POST`, `request.FILES` y rastrean hacia funciones peligrosas.

```bash
pip install semgrep

# Escanear con reglas específicas para Django
semgrep --config "p/django" ./miproyecto

# Reglas de seguridad de OWASP
semgrep --config "p/owasp-top-ten" ./miproyecto
```

> *Fuente: Semgrep. (2024). Django security rules.* [https://semgrep.dev/p/django](https://semgrep.dev/p/django)

### 5.2 De padre a hijo (dirección inversa — el poder real)

Desde una instancia de `Categoria`, podemos acceder a todos sus productos. Aquí es donde `related_name` hace la diferencia:

```python
# Sin related_name definido → Django genera el nombre automáticamente
categoria.producto_set.all()    # Sintaxis poco expresiva

# Con related_name='productos' → sintaxis que describe el negocio
categoria.productos.all()       # Mucho más claro
```

Todos los métodos de QuerySet funcionan igual en la relación inversa:

```python
telescopios = Categoria.objects.get(nombre='Telescopios')

# Todos los productos de esa categoría
todos        = telescopios.productos.all()

# Filtrar dentro de esa categoría
en_stock     = telescopios.productos.filter(stock__gt=0)
mas_caros    = telescopios.productos.filter(precio__gte=500).order_by('-precio')

# Contar (una sola consulta SQL — no carga los objetos)
cuantos      = telescopios.productos.count()

# Verificar si tiene alguno (más eficiente que count)
tiene_alguno = telescopios.productos.exists()
```

### 5.3 Filtros que cruzan la relación con `__`

El doble guión bajo `__` permite cruzar relaciones en cualquier consulta:

```python
# Productos de la categoría "Telescopios"
Producto.objects.filter(categoria__nombre='Telescopios')

# Productos de categorías cuyo nombre empieza con "T" (sin importar mayúsculas)
Producto.objects.filter(categoria__nombre__istartswith='T')

# Categorías que tienen al menos un producto en stock
Categoria.objects.filter(productos__stock__gt=0).distinct()
# → .distinct() porque sin él aparece una fila por cada producto encontrado

# Categorías que tienen al menos un producto activo y costoso
Categoria.objects.filter(
    productos__activo=True,
    productos__precio__gte=1000
).distinct()
```

> **¿Por qué `distinct()`?** Cuando cruzamos una relación inversa (de Categoria hacia Producto), SQL genera una fila por cada coincidencia. Si la categoría "Telescopios" tiene 5 productos en stock, aparecería 5 veces en el resultado. `distinct()` elimina los duplicados.
>
> *Fuente: Django Documentation. (2024). Making queries — Spanning relationships.*

### 5.4 related_name en relaciones múltiples al mismo modelo

Si un modelo tiene **dos o más `ForeignKey` apuntando al mismo modelo**, es obligatorio definir `related_name` diferente para cada una, o usar `related_name='+'` para desactivar la relación inversa:

```python
class Envio(models.Model):
    origen  = models.ForeignKey(
        'Ciudad',
        on_delete=models.PROTECT,
        related_name='envios_salientes',  # ciudad.envios_salientes.all()
    )
    destino = models.ForeignKey(
        'Ciudad',
        on_delete=models.PROTECT,
        related_name='envios_entrantes',  # ciudad.envios_entrantes.all()
    )
```

Sin `related_name` diferente, Django levantará este error:

```
django.core.exceptions.FieldError:
Reverse accessor for 'Envio.origen' clashes with reverse accessor
for 'Envio.destino'. Add or change a related_name argument to the
definition for 'Envio.origen' or 'Envio.destino'.
```
